# RAG 全流程生产级优化策略总览

> 按**索引 → 检索 → 生成**三阶段，系统梳理每个环节的优化手段，并列举具体技术。

---

## 一、索引阶段
**目标：把原始文档变成可被高效、精准检索的向量和元数据。**

### 1.1 文档解析与清洗
- **格式解析**：PDF/Word → Markdown/TXT，保留标题、表格、层级结构。
- **元数据提取**：标题、作者、日期、章节、页数、来源等。
- **文本清洗**：去页眉页脚、乱码、HTML标签、多余空白。
- **OCR预处理**：扫描件/图片的文字识别。

### 1.2 智能分块
- **递归分块**：按 `\n` → `。` → `，` → 空格 → 逐字 的优先级切分。
- **结构分块**：按Markdown标题、PDF章节标记切分。
- **语义分块**：计算相邻句子Embedding相似度，在语义骤降处切分。
- **Agentic分块**：大模型主动识别断点，并预生成块标题、假设问题。
- **Late Chunking**：先对整篇文档做Attention，再进行Token级向量化与分块。
- **固定大小分块（含重叠）**：简单控制 `chunk_size` 与 `chunk_overlap`。

### 1.3 分块参数调优
- `chunk_size`：通常 256~1024 tokens（小则丢上下文，大则引入噪声）
- `chunk_overlap`：chunk_size 的 10%~20%
- 分隔符优先级自定义

### 1.4 向量化与索引策略
- **单向量索引**：每个块生成一个 Embedding 向量。
- **多向量索引**：同一文档存多个向量（原文、摘要、关键词、假设问题等）。
- **父-子文档索引**：小块向量检索，返回大块完整上下文。
- **ColBERT 级 Token 索引**：每个 Token 单独编码，检索时进行延迟交互匹配。
- **元数据 Schema 设计**：为每块标注类别、来源、时间、对象等结构化字段。

### 1.5 嵌入模型选择与微调
- **通用模型**：`BAAI/bge-large-zh-v1.5`、`text-embedding-3-large`、`Jina embeddings v3`
- **领域微调**：使用心理学等垂直领域数据微调 Embedding 模型。
- **对比学习**：`MultipleNegativesRankingLoss`、`TripletLoss`、`CosineSimilarityLoss`
- **Matryoshka 表示**：支持多维度输出，推理时可灵活控制向量维度。
- **CLS/Mean pooling 选择**：根据模型调整池化策略。

### 1.6 向量数据库选型
- **Milvus**：分布式、高性能、支持混合检索与丰富过滤
- **Chroma**：轻量、易集成
- **Weaviate**：自带向量+关键词混合检索
- **Pinecone**：全托管，免运维
- **Qdrant**：高性能，过滤能力强
- **Elasticsearch + 向量插件**：传统检索+向量检索融合

## 二、检索阶段
**目标：将用户自然语言转变为精准的检索动作，召回所有可能的相关文档。**

### 2.1 查询重写（核心优化）
把用户模糊、简短、口语化、多意图的查询转化为一个或多个更适合检索的查询。

#### 2.1.1 Multi-Query（多查询改写）
- **原理**：用大模型把同一个原始问题改写成多个语义相近但表述不同的查询。
- **作用**：从不同角度召回结果，提高覆盖率。
- **技术**：Prompt 工程引导大模型生成多条并行查询。

#### 2.1.2 Query Decomposition（问题分解）
- **原理**：将复杂复合问题拆解成多个独立的子问题，依次检索，最终合并上下文。
- **适用**：“怎么缓解压力和失眠？”→ “如何缓解压力？”“如何改善失眠？”
- **技术**：`Least-to-Most Prompting`、`IRCoT`、`ReAct` 式分解。

#### 2.1.3 Step-back Prompting（后退式提示 / 抽象化）
- **原理**：先让LLM围绕原问题生成一个更高层次、更宽泛的概念问题，用概念问题去检索，然后结合原问题回答。
- **例子**：原问题“GAD-7量表评分标准是什么？” → 后退问题“焦虑障碍常用筛查工具有哪些？”
- **技术**：通过特殊提示词要求大模型生成“step-back question”。

#### 2.1.4 HyDE（Hypothetical Document Embeddings）
- **原理**：让大模型根据用户问题**生成一个假想的答案文档**，然后用该假想文档的向量去检索。
- **作用**：将“问题-文档”匹配转变为“文档-文档”匹配，解决短查询语义稀疏问题。
- **技术**：`Prompt: "Write a paragraph that answers the question..."` 然后用生成文本做向量检索。

#### 2.1.5 自查询检索器（Self-Querying Retriever）
- **原理**：LLM 自动将用户自然语言解析为 `{query, filter}` 结构，同时完成语义搜索和元数据过滤。
- **技术**：定义 `AttributeInfo` 描述元数据字段，LLM 提取过滤条件（`eq`, `gt`, `lt`, `and`, `or`）。

#### 2.1.6 上下文补全
- **场景**：指代消解、历史信息补充，多轮对话中补全“它有什么副作用？”为“舍曲林有什么副作用？”
- **技术**：用 LLM 结合对话历史重写当前查询。

#### 2.1.7 假设性问题嵌入（Hypothetical Questions）
- **原理**：离线阶段用 LLM 为每篇文档生成几个它能回答的问题，将这些问题的向量存入索引；在线查询直接匹配这些问题，再返回原文档。
- **本质**：查询重写的一种“预处理”形态，拉近查询与文档的语义距离。

#### 2.1.8 Query2Doc / 伪相关反馈
- **技术**：LLM 根据原始查询生成少量相关伪文档，结合伪文档做查询扩展或直接检索。

### 2.2 多路召回
- **向量语义召回**：稠密向量相似度检索。
- **关键词全文召回**：BM25、Elasticsearch 精确匹配。
- **元数据过滤**：按类别、时间、来源等硬性条件筛选。
- **知识图谱检索**：实体关系查询，医疗/法律等强关联场景。
- **稀疏向量召回**：SPLADE 等学习型稀疏向量。

### 2.3 融合排序
- **RRF（倒数排名融合）**：无视分数量纲，仅根据排名位置计算融合分数。
- **加权分数融合**：归一化后按预设权重求和。
- **CombSUM / CombMNZ**：分数求和，或求和×出现次数。
- **学习型融合**：用小型排序模型决定最终顺序。

### 2.4 检索路由
- **查询意图分类**：情感支持类走语义检索，事实知识类走混合检索，工具调用类直接路由。
- **动态检索深度**：简单问题 Top-3，复杂问题 Top-10。
- **多索引路由**：根据问题领域（心理学/法律）路由到对应知识库索引。

## 三、增强阶段（检索后 → 生成前）
**目标：对召回文档进行精细加工，把最有价值的信息以最高效的形式递给大模型。**

### 3.1 重排序（Reranking）
- **Cross-Encoder 重排**：对 (query, doc) 联合编码打分，精度远高于双塔。常用模型：`BAAI/bge-reranker`、`Cohere Rerank`。
- **领域专用 Reranker**：用垂直领域数据微调 Cross-Encoder，使其更懂行业语义。
- **多维度重排**：综合相关性、时效性、权威性、共情度等加权打分。
- **LLM 作为 Reranker**：大模型直接对文档列表打分或排序，精度高但延迟大、成本高。
- **学习型排序**：LambdaMART、ListNet 等 LTR 模型，适合有大量点击反馈的场景。

### 3.2 上下文压缩
- **提取式压缩**：用 Cross-Encoder 对文档内每个句子打分，仅保留 Top-N 相关句。
- **摘要式压缩**：用轻量大模型对文档进行有损摘要，极大缩短长度。
- **递归压缩（Map-Reduce）**：对大量文档先分组摘要，再对摘要做摘要。
- **自适应压缩**：根据查询类型动态调整压缩率（情感类保留更多细节）。
- **LLMLingua 等专用压缩算法**：小型语言模型对 Prompt 进行 token 级剪枝。

### 3.3 Prompt 构建
- **分层 Prompt 设计**：`[系统指令] → [核心文档全文] → [补充文档压缩版] → [对话历史] → [用户问题]`。
- **黄金位置原则**：最相关内容放在 Prompt 头部，利用注意力偏置。
- **引用标注**：为每段文档添加 `[来源: xxxx]`，便于溯源和验证。
- **Token 预算控制**：文档部分不超过上下文窗口的 30%~40%，留足生成空间。
- **结构化输入格式**：使用 XML/JSON 标记分隔不同块，提高模型信息定位能力。

## 四、生成阶段
**目标：基于增强上下文生成高质量、安全、可信的回答。**

### 4.1 Prompt 工程
- **System Prompt 精调**：明确角色、语调、知识边界、安全红线。
- **Few-shot 示例**：在 Prompt 中给出 2-3 个理想回复样例。
- **思维链（Chain-of-Thought）**：引导模型分步推理再回答。
- **结构化输出**：要求输出 JSON、Markdown 表格等，便于后处理。
- **拒绝回答模板**：当信息不足或高风险时，引导模型固定话术回复。

### 4.2 生成控制
- **温度与 Top-p**：事实类低温度 (0~0.3)，创意类略高。
- **最大 Token 限制**：防止生成过长或截断。
- **流式输出**：逐 token 返回，提升用户体验。
- **重复惩罚**：避免模型陷入循环。

### 4.3 安全与质量保障
- **内容安全审查**：检测自伤、暴力、违法内容，使用 Guardrails 或专用模型。
- **幻觉检测**：用 NLI 模型检查生成内容是否可被检索文档支持（如 `BAAI/bge-reranker` 可做蕴含判断）。
- **拒识机制**：当检索置信度低于阈值时，明确告知“我目前的知识无法回答此问题”。
- **安全路由**：高危查询直接拦截并回复固定话术或转人工。
- **事实一致性验证**：关键数字、日期、实体进行规则或小模型二次校验。

### 4.4 后处理
- **格式修复**：修正 Markdown/JSON 格式错误。
- **敏感信息脱敏**：正则移除或替换手机号、身份证、地址等 PII。
- **引用补全与对齐**：确保声明的事实都能对应到具体来源编号。
- **输出拦截**：再次过滤不合规内容。

## 五、持续优化与监控
- **离线评估**：检索指标 `Recall@K`、`MRR`、`NDCG@K`；生成指标 `ROUGE-L`、`bert-score`、人工评估。
- **Bad Case 分析**：定期分析“搜不准”“答不好”的案例，针对性优化分块、检索或重排。
- **A/B 测试**：新旧方案对比，用线上真实数据验证效果。
- **监控告警**：延迟 P95/P99、检索召回率、生成拒绝率、用户负反馈率。
- **数据飞轮**：收集用户纠错、点赞/踩等反馈，转换为新的训练数据（微调模型或调整检索策略）。
